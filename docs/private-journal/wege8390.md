# Quiz Controller Iteration Journal

## Iteration 1: Fix question count — random 6–10 → fixed 10

- **Commits:** `fix: set quiz length to fixed 10 questions`
- **What:** Replaced `Math.floor(Math.random() * 5) + 6` with a fixed `size: 10` in the `$sample` aggregation stage. Added a guard `if (questions.length < 10)` to return an error when the active question pool is insufficient.
- **Challenge:** Initially used random 6–10 per the spec's wording, but this caused leaderboard fairness issues — a user answering 6 questions and scoring 6/6 is not comparable to another scoring 8/10. The spec's "6–10" is developer flexibility, not a player-facing feature.
- **How solved:** After group discussion, decided on fixed 10 for all attempts. This ensures scores are directly comparable on the leaderboard and simplifies frontend state management (`currentIndex` always counts to 10). Documented this decision in README per spec §7.3.

## Iteration 2: Remove correctAnswer and explanation from startQuiz response

- **Commits:** `fix: remove correctAnswer from quiz start API response`
- **What:** The original `$project` stage included `explanation` alongside `questionText` and `options`. Removed `explanation` from the start endpoint projection so that only `questionText` and `options` are sent to the client during the quiz. The `correctAnswer` field was already excluded.
- **Challenge:** Exposing `explanation` in the start response leaks information — a user inspecting the network tab could read the explanation (which often hints at the correct answer) before submitting. This is a security concern under the spec's "robustness and correctness" emphasis.
- **How solved:** Restricted the `$project` to `{ questionText: 1, options: 1 }` only. The `explanation` and `correctAnswer` are now only returned in the submit response as part of the review data, after the user has already locked in all answers.

## Iteration 3: Fix N+1 query in submitQuiz

- **Commits:** `perf: batch fetch questions in submitQuiz to fix N+1`
- **What:** The original implementation called `Question.findById(ans.questionId)` inside a `for` loop, resulting in 10 separate database queries. Refactored to a single `Question.find({ _id: { $in: questionIds } })` call with a hashmap lookup.
- **Challenge:** The N+1 pattern works during development with a local MongoDB instance, but under load (or with network latency to MongoDB Atlas), 10 sequential queries add up. This also made the function harder to reason about — each iteration was both querying and computing.
- **How solved:** Collected all `questionId` values upfront with `answers.map(a => a.questionId)`, fetched all questions in one query, built a `questionMap` keyed by `_id.toString()`, then iterated over answers using map lookups only. Reduced database round trips from 10 to 1.

## Iteration 4: Add input validation for submitQuiz

- **Commits:** `feat: add comprehensive input validation for quiz submission`
- **What:** Added three validation checks before processing: (1) `answers.length !== 10` rejects partial or oversized submissions, (2) duplicate `questionId` detection via `Set`, (3) each `selectedAnswer` must be an integer between 0 and 3.
- **Challenge:** Without the length check, a user could submit only the questions they got right (e.g., 3/3 = 100%) while ignoring the other 7. The original code silently accepted any number of answers and any `selectedAnswer` value.
- **How solved:** Added early-return validation at the top of `submitQuiz`. Used `Number.isInteger()` instead of just `typeof === 'number'` to reject values like `1.5`. Each validation failure returns a specific error message through the `fail()` envelope so the frontend can display meaningful feedback.

## Iteration 5: Return review data for Review Mode variation

- **Commits:** `feat: return full review data in submit response for review mode`
- **What:** Extended the submit response from `{ score, scoreId }` to include a `review` array containing `questionText`, `options`, `selectedAnswer`, `correctAnswer`, `isCorrect`, and `explanation` for each question. This is the data the frontend needs to render the Review Mode page immediately after quiz completion.
- **Challenge:** The original response only returned `detailedAnswers` (questionId + selectedAnswer + isCorrect), which is what gets stored in the Score model. But the Review Mode UI needs the full question text and explanation — data that lives in the Question model, not the Score model. Deciding where to assemble this join was the key design question.
- **How solved:** Assembled the review data server-side by mapping over `detailedAnswers` and enriching each entry from the `questionMap` that was already loaded for scoring. This avoids an extra API call from the frontend and keeps the review data consistent with the scoring logic. The Score model still stores only the minimal `{ questionId, selectedAnswer, isCorrect }` — the review data is computed on the fly, not duplicated in storage.

## Iteration 6: Add history detail endpoint and leaderboard username

- **Commits:** `feat: add attempt detail endpoint and leaderboard with username`
- **What:** Added `getAttemptDetail` (GET `/api/quiz/history/:id`) to support re-reviewing past attempts from the history page. Updated `getLeaderboard` to use `$lookup` on the users collection so the response includes `username` instead of raw `userId`. Updated `getHistory` to select only `score` and `createdAt` for the list view.
- **Challenge:** For `getAttemptDetail`, a question might have been deleted by an admin after the user took the quiz. Accessing `q.text` on a null object would crash the server. For the leaderboard, the original `$group` + `$sort` pipeline returned only `_id` (userId) and `bestScore` — useless for the frontend without a username.
- **How solved:** Added a null check in `getAttemptDetail` — if a question no longer exists, the review entry shows `"[Question deleted]"` with `correctAnswer: null` instead of throwing. For the leaderboard, appended `$lookup` → `$unwind` → `$project` stages to join with the users collection. Also added ownership validation (`userId: req.user.id`) to prevent users from viewing other users' attempt details.
